# Farhan Thawar

**Vice President and Head of Engineering** | **Shopify** | Expertise: engineering-management, leadership, hiring-culture, operations

## Bio
Farhan Thawar is the VP and Head of Engineering at Shopify, where he helps shape a remote-first engineering organization built around speed, simplicity, and craft. His perspective matters because he combines deep operational rigor with unusually strong opinions on how teams should learn, hire, collaborate, and delete complexity.

He stands out for treating engineering as a high-leverage management system: optimize for learning, use real work to evaluate talent, and keep codebases and calendars aggressively lean. If you want a view on how to run a large engineering org without losing intensity or quality, Farhan is a strong voice.

## Signature Frameworks

### Choose the Hard Path
- **When to use**: When choosing between an easy option and a harder one that may teach more.
- **Steps**:
  1. Identify the easy path and the harder path.
  2. Ask which option creates the most learning and exposure to strong people.
  3. Prefer the path that still leaves you better off even if it fails.
  4. Use the experience to build durable skills and confidence in ambiguity.
- **Example**: Instead of sending out resumes all day, build an app or extension for the company you want to join. Even if you do not get hired, you gain portfolio work and domain knowledge.

### Personal Job-Selection Framework
- **When to use**: When deciding whether to take, stay in, or leave a role.
- **Steps**:
  1. Write down what matters most: learning, impact, people, growth, compensation, etc.
  2. Evaluate every opportunity against that list.
  3. Revisit the framework as your priorities change.
  4. Leave decisively when a role no longer fits.
- **Example**: Farhan has left roles when the work stopped matching his values, including after an acquisition changed the nature of the job.

### Pair Programming as a Management Tool
- **When to use**: For critical engineering problems, onboarding, incidents, and knowledge transfer.
- **Steps**:
  1. Put two engineers on one problem.
  2. Focus on reasoning and design, not just typing faster.
  3. Use the session to improve quality and confidence.
  4. Apply it selectively where the leverage is highest.
- **Example**: Shopify often pairs for 4–8 hours per week, and historically Toby and the CTO would pair for an hour and delete the code if the feature was not done, keeping only the tests.

### Delete-and-Rebuild for Elegant Solutions
- **When to use**: When code has become complex, brittle, or clearly wrong.
- **Steps**:
  1. Build enough to understand the problem.
  2. Judge whether the implementation is truly simple.
  3. Delete aggressively if the design is off.
  4. Rebuild with the new understanding.
- **Example**: Shopify’s “Delete Code Club” and hack days encourage teams to remove large amounts of code and simplify systems.

### GSD + Six-Week Review Cadence
- **When to use**: When you need urgency, alignment, and fast feedback in a large org.
- **Steps**:
  1. Require weekly updates.
  2. Use layered checkpoints for leadership alignment.
  3. Run six-week reviews on roadmap, resourcing, and blockers.
  4. Act immediately on new information.
- **Example**: A team blocked by LLM context limits got unblocked during a review when Farhan asked model providers directly, leading to a solution within an hour.

### Experiment vs Feature vs Infrastructure
- **When to use**: When deciding how to scope technical work.
- **Steps**:
  1. Classify the work as an experiment, feature, or infrastructure.
  2. Optimize experiments for learning.
  3. Build features on existing infrastructure when possible.
  4. Invest more in infrastructure when it will compound future speed.
- **Example**: For NFT gating, the better move was not just a one-off feature, but a platform layer that could support many future use cases.

### Meeting Armageddon
- **When to use**: When recurring meetings have taken over maker time.
- **Steps**:
  1. Delete recurring internal meetings with more than two people once a year.
  2. Keep only one-on-ones, interviews, and external partner meetings.
  3. Pause new recurring meetings temporarily.
  4. Recreate only what is truly needed.
- **Example**: Shopify’s annual reset reduced individual contributor meeting time from roughly 5–6 hours per week to about 3.

### Trust Battery
- **When to use**: When managing remote teams and calibrating autonomy.
- **Steps**:
  1. Treat trust as something that can be charged or depleted.
  2. Recharge it with in-person summits or bursts.
  3. Adjust autonomy and communication based on trust level.
  4. Restore trust proactively when remote work drains it.
- **Example**: Shopify uses company-wide summits and smaller team bursts to rebuild trust in a mostly remote environment.

## Core Advice
- **When choosing between easy and hard**: pick the harder path — because even failure leaves you with skills, relationships, and insight.
- **When job searching**: build something real instead of only applying — because proof of work beats polished intent.
- **When unsure about a role**: write a personal decision framework — because titles and brand can distract from what you actually value.
- **When afraid to ask a question**: ask it anyway — because many others are confused too, and learning matters more than looking smart.
- **When engineering work is critical**: pair program — because the bottleneck is usually solution quality, not typing speed.
- **When code is getting messy**: delete and rebuild — because code is a liability, and simplicity compounds.
- **When execution is drifting**: use weekly updates and six-week reviews — because frequent checkpoints expose blockers early.
- **When meetings are multiplying**: delete recurring meetings in bulk — because calendar inertia quietly destroys maker time.
- **When a feature request appears**: ask what infrastructure would make it trivial — because platform work compounds.
- **When hiring**: use real work trials where possible — because interviews are weak predictors of actual performance.
- **When remote teams feel loose**: create intentional in-person moments — because trust needs periodic recharging.
- **When everything feels too calm**: take on a harder bet — because some discomfort is a sign of real learning and ambition.

## Contrarian Takes
- **Conventional**: Work harder means more hours. → **Their view**: Do more per minute. — because intensity and focus beat raw time.
- **Conventional**: Pair programming halves output. → **Their view**: Pairing often increases throughput. — because the real bottleneck is finding the right solution.
- **Conventional**: More meetings always slow teams down. → **Their view**: Well-designed check-ins can speed teams up. — because they create urgency and unblock work.
- **Conventional**: Micromanagement is always bad. → **Their view**: Close involvement can be useful if it is collaborative. — because leaders can add context and solve problems directly.
- **Conventional**: Shipping more code equals more productivity. → **Their view**: Deleting code is often better. — because simpler systems are faster and safer.
- **Conventional**: Build the requested feature directly. → **Their view**: Build infrastructure that makes the feature easy later. — because platform investments compound.
- **Conventional**: Interviews are the best hiring filter. → **Their view**: Real work is a better signal. — because it reveals actual performance and collaboration.
- **Conventional**: Comfort creates security. → **Their view**: Discomfort and learning create real stability. — because adaptability is the durable advantage.
