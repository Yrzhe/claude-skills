# Farhan Thawar

**Vice President and Head of Engineering** | **Shopify** | Expertise: engineering-management, leadership, hiring-culture, operations, strategy

## Bio
Farhan Thawar leads engineering at Shopify and helps shape one of the most intense, craft-oriented, remote-first engineering cultures in tech. His perspective matters because he combines operator rigor with unusually strong opinions on how teams should learn, hire, build, and stay fast over time.

He is especially valuable for leaders who want practical ways to improve engineering quality without adding bureaucracy. Farhan’s thinking is rooted in real systems: pair programming, deleting code, hiring through actual work, and using cadence to keep large organizations moving.

## Signature Frameworks

### Choose the Hard Path
- **When to use**: When deciding between options and one is harder but offers more learning, exposure, or durable growth.
- **Steps**:
  1. Identify the easy path and the harder path.
  2. Compare them on learning, people, and skill development.
  3. Prefer the path that still leaves you better off even if it fails.
  4. Use the experience to build confidence in ambiguity.
- **Example**: Instead of sending resumes all day, build an app or extension for the company you want to join. Even if you don’t get hired, you gain portfolio work and domain knowledge.

### Personal Job-Selection Framework
- **When to use**: When choosing a job, deciding whether to stay, or evaluating a career move.
- **Steps**:
  1. Write down what matters most: learning, impact, people, growth, etc.
  2. Evaluate every opportunity against that list.
  3. Ignore status signals like title, brand, or compensation when they conflict with your values.
  4. Revisit the framework as your priorities change.
- **Example**: Farhan has left roles when they no longer matched his framework, including after an acquisition changed the nature of his work.

### Pair Programming as a Management Tool
- **When to use**: For critical engineering problems, onboarding, incidents, knowledge transfer, and quality-sensitive work.
- **Steps**:
  1. Put two engineers in one shared problem-solving session.
  2. Focus on reasoning and design, not just typing speed.
  3. Use the session to improve clarity and catch bad ideas early.
  4. Pair selectively where the leverage is highest.
- **Example**: Shopify uses pairing for a few hours per week, and historically Toby and the CTO would pair for an hour and delete all code if the feature wasn’t done, keeping only tests.

### Delete-and-Rebuild for Elegant Solutions
- **When to use**: When code has become complex, brittle, or was built before the team understood the problem.
- **Steps**:
  1. Build enough to understand the domain.
  2. Judge whether the implementation is truly simple.
  3. Delete code aggressively if the design is wrong.
  4. Rebuild with the new understanding.
- **Example**: Shopify runs “Delete Code Club” and hack days focused on removing large amounts of code to keep systems clean.

### GSD + Six-Week Review Cadence
- **When to use**: When you need urgency, alignment, and fast feedback in a large org.
- **Steps**:
  1. Require weekly project updates.
  2. Use layered checkpoints like OK1 and OK2.
  3. Run six-week leadership reviews.
  4. Act on blockers immediately.
- **Example**: A team blocked by LLM context limits got unblocked during a review when Farhan asked model providers directly and expanded the context window within an hour.

### Experiment vs Feature vs Infrastructure
- **When to use**: When deciding how to scope technical work.
- **Steps**:
  1. Classify the work as an experiment, feature, or infrastructure.
  2. Optimize experiments for learning.
  3. Build features on existing infrastructure when possible.
  4. Invest more in infrastructure when it unlocks many future features.
- **Example**: Instead of building NFT gating as a one-off feature, Shopify built a platform layer so others could create similar use cases quickly.

### Meeting Armageddon
- **When to use**: When recurring meetings have accumulated and are stealing maker time.
- **Steps**:
  1. Delete all internal recurring meetings with more than two people once a year.
  2. Keep only one-on-ones, interviews, and external partner meetings.
  3. Freeze new recurring meetings temporarily.
  4. Recreate only the meetings that prove necessary.
- **Example**: Shopify’s annual reset reduced IC meeting time from roughly 5–6 hours per week to about 3.

### Trust Battery
- **When to use**: When managing remote teams and calibrating autonomy.
- **Steps**:
  1. Treat trust as something that can be charged or depleted.
  2. Recharge it through in-person summits or bursts.
  3. Adjust autonomy and communication based on trust level.
  4. Restore trust proactively before it decays.
- **Example**: Shopify uses company-wide summits and smaller team bursts to recharge trust in a mostly remote environment.

## Core Advice
- **When choosing between an easy path and a hard one**: take the hard path — because even failure leaves you with skills, relationships, and confidence.
- **When job searching by sending resumes**: build something real instead — because proof of work beats polished claims.
- **When unsure what job to take**: write a personal decision framework — because it prevents you from optimizing for the wrong signals.
- **When you’re afraid to ask a basic question**: ask it anyway — because understanding matters more than looking smart.
- **When engineering work feels slow or siloed**: pair on the problem — because the bottleneck is usually solution quality, not typing speed.
- **When a codebase is getting messy**: delete and rebuild — because code is a liability if it’s not simple.
- **When execution is drifting**: add weekly updates and six-week reviews — because frequent checkpoints expose blockers early.
- **When recurring meetings pile up**: delete them in bulk — because meeting inertia is real.
- **When a feature request appears on a platform**: ask what infrastructure would make it trivial — because platform work compounds.
- **When hiring relies mostly on interviews**: use real work trials where possible — because interviews are weak predictors.
- **When remote teams start to feel disconnected**: create intentional in-person moments — because trust needs recharging.
- **When everything feels too calm**: take on a few uncomfortable bets — because growth often lives near the edge of chaos.

## Contrarian Takes
- **Conventional**: Working harder means more hours. → **Their view**: Do more per minute, not more hours — because intensity and focus matter more than time spent.
- **Conventional**: Pair programming cuts output in half. → **Their view**: Pairing often increases throughput — because the real bottleneck is finding the right solution.
- **Conventional**: More meetings always slow teams down. → **Their view**: Well-designed check-ins can speed teams up — because they create urgency and unblock work.
- **Conventional**: Micromanagement is always bad. → **Their view**: Close involvement can be useful if it’s collaborative problem-solving — because context and pairing help.
- **Conventional**: Shipping more code equals more productivity. → **Their view**: Deleting code is often a better sign of maturity — because simpler systems move faster.
- **Conventional**: Build the requested feature directly. → **Their view**: Build infrastructure that makes the feature easy later — because compounding platform value beats one-offs.
- **Conventional**: Interviews are the best hiring filter. → **Their view**: Real work is a better signal — because it reveals actual performance and collaboration.
- **Conventional**: Comfort creates security. → **Their view**: Discomfort and learning create real stability — because adaptability is the durable advantage.
