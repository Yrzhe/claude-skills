## Signature Frameworks

### Context-first prompting
- **When to use**: When outputs feel generic, shallow, or too dependent on the model’s default assumptions.
- **Steps**:
  1. Explain the task, audience, and desired outcome.
  2. Add domain-specific details, constraints, and examples.
  3. Attach source material like docs, links, tweets, or prior work.
  4. Iterate by adding more context until the answer becomes nuanced.
  5. Use the model to draft, then edit for polish.
- **Example**: For interview prep, give ChatGPT links to your blog and Twitter so it can generate questions based on your actual thinking instead of generic prompts.

### GPTs as packaged workflows
- **When to use**: When a team repeatedly does the same task and you want a reusable AI experience for non-technical users.
- **Steps**:
  1. Identify a narrow, repeatable problem.
  2. Encode the context in custom instructions.
  3. Upload files or knowledge sources.
  4. Add tools like browsing, code interpreter, image generation, or APIs.
  5. Share the GPT internally or publicly.
- **Example**: A GPT that writes Facebook and Google ads, or one that summarizes experiment results and answers follow-up questions.

### Build for the general model, differentiate in the vertical
- **When to use**: When deciding whether to build a startup on OpenAI APIs or compete with OpenAI directly.
- **Steps**:
  1. Use OpenAI for broad capabilities like reasoning, coding, and writing.
  2. Add domain data, fine-tuning, and workflow design for a specific vertical.
  3. Build custom UI and product features OpenAI is unlikely to prioritize.
  4. Avoid generic assistants unless they are radically better.
  5. Focus on a customer problem where domain knowledge creates an edge.
- **Example**: Harvey as a legal AI product that can outperform a general assistant in its niche.

### High-agency, high-urgency operating model
- **When to use**: When building a fast-moving team or hiring for one.
- **Steps**:
  1. Hire people who identify problems independently.
  2. Prefer people who act without waiting for consensus.
  3. Give direct customer-to-solution ownership.
  4. Let trusted people move quickly with fewer approvals.
  5. Use real-time coordination tools to keep execution tight.
- **Example**: The assistants API emerged because the team saw repeated developer demand and quickly aligned to ship a higher-level abstraction.

### Measure progress through proxies, not just the headline goal
- **When to use**: When working toward a long-term mission that can’t be measured directly in the short term.
- **Steps**:
  1. Define the ultimate mission clearly.
  2. Choose intermediate metrics that reflect progress.
  3. Use revenue, adoption, and developer growth as proxies when appropriate.
  4. Re-evaluate metrics as conditions change.
  5. Don’t confuse the proxy with the real goal.
- **Example**: Revenue matters because it funds compute and GPUs, which help OpenAI train better models and move toward AGI.

## Core Advice
- **When you face a broad, generic AI use case**: Don’t build a generic assistant unless it is radically better than ChatGPT — because OpenAI will keep improving the general product.
- **When you want AI to solve an internal company task**: Package it as a GPT with instructions, files, and tools — because reusable workflows beat repeated ad hoc prompting.
- **When you need better prompt results**: Give the model more context than you think it needs — because richer inputs produce more specific, useful outputs.
- **When you’re prompting for nuanced work**: Feed source material instead of relying on built-in knowledge — because the model may not know enough about your exact topic.
- **When you want slightly better model behavior**: Try small cues like style instructions or even playful prompts — because models respond to human-like communication patterns.
- **When you’re building on OpenAI APIs**: Focus on a vertical or workflow with domain expertise — because specialization is the best defense against being displaced.
- **When chat is the default interface**: Move beyond chat if the task deserves a better UX — because canvas, workflow, or action-oriented interfaces can unlock more value.
- **When you want company-wide AI gains**: Start with engineering, planning, marketing, and experiment analysis — because these functions are already strong fits for AI.
- **When planning a project or quarter**: Use a GPT to force rigor around timelines, metrics, and stakeholders — because it can surface dependencies humans miss.
- **When hiring for a fast-moving AI team**: Prioritize high agency and urgency — because speed and independence reduce bottlenecks.
- **When rolling out ChatGPT in a company**: Use Teams or Enterprise — because centralized access, security, and shared GPTs make adoption safer and more useful.
- **When you find a bug in ChatGPT**: Send a reproducible example with the exact prompt — because vague complaints are hard to debug.

## Contrarian Takes
- **Conventional**: AI startups should avoid anything OpenAI might ship → **Their view**: Vertical products with domain expertise can still win — because OpenAI focuses on general capabilities, not niche workflows.
- **Conventional**: The best AI products will be generic assistants → **Their view**: Generic assistants are the hardest category to win — because they compete directly with ChatGPT.
- **Conventional**: Prompt engineering is about clever hacks → **Their view**: Prompt engineering is about context — because better inputs matter more than tricks.
- **Conventional**: AI will replace humans outright → **Their view**: Humans using AI will replace humans who don’t — because augmentation creates the real productivity edge.
- **Conventional**: More researchers always means more output → **Their view**: In GPU-constrained environments, more researchers can reduce productivity — because compute bottlenecks matter.
- **Conventional**: Revenue should be the main goal → **Their view**: Revenue is often a proxy — because it enables compute, which advances the real mission.
